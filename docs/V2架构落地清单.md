# 招聘筛选系统 V2 架构落地清单（Recommend 闭环）

## 1. 目标与边界

### 1.1 V2 目标
- 固化核心闭环：`BOSS 推荐牛人 -> 抓取简历 -> 结构化 -> LLM+评分卡匹配 -> 自动打招呼 -> HR复核`
- 全链路可审计：每个候选人的来源、证据、模型调用、打分、动作可追踪
- 可运营：任务成功率、打招呼成功率、Token 成本、模型回退率可量化

### 1.2 范围（In Scope）
- 仅支持 `recommend` 流程
- 模型统一使用 `kimi-for-coding`（兼容后续切换）
- 本地 Playwright + `storage_state` 会话复用
- HR 后台：创建任务、查看 checklist、复核

### 1.3 非目标（Out of Scope）
- 不做多平台采集（仅 BOSS）
- 不做自动改聊术生成（保留固定打招呼动作）
- 不做复杂 ATS/CRM 双向同步（V2 仅预留接口）

---

## 2. 总体架构（逻辑分层）

### 2.1 组件
- `Collector`：Playwright 页面自动化（列表扫描、详情打开、下载简历、截图）
- `Extractor`：LLM 字段抽取（HR Prompt + JSON 输出约束）
- `Scorer`：评分卡引擎（硬性条件 + 权重 + 阈值）
- `Action Executor`：自动打招呼/跳过/失败重试
- `Review Console`：HR Checklist + 复核回写
- `Observability`：运行日志、指标聚合、告警

### 2.2 核心原则
- LLM 负责“语义抽取与解释”，规则引擎负责“最终决策”
- 任何自动动作都要落库（动作前条件 + 结果）
- 失败可恢复：任务可重跑、候选人可重试、日志可定位

## 3.3 V2 新增表

### `model_calls`（模型调用审计）
```sql
create table if not exists model_calls (
  id text primary key,
  task_id text not null,
  candidate_id text,
  stage text not null,                 -- extract / score_reason / summary
  provider text not null,              -- kimi_cli / openai_compatible
  model text not null,
  prompt_tokens integer not null default 0,
  completion_tokens integer not null default 0,
  total_tokens integer not null default 0,
  latency_ms integer not null default 0,
  status text not null,                -- success / failed / timeout / fallback
  error_message text,
  created_at text not null default current_timestamp
);
create index if not exists idx_model_calls_task on model_calls(task_id, created_at);
create index if not exists idx_model_calls_candidate on model_calls(candidate_id, created_at);
```

### `boss_session_checks`（会话健康检查）
```sql
create table if not exists boss_session_checks (
  id text primary key,
  status text not null,                -- valid / invalid
  reason text,
  current_url text,
  screenshot_path text,
  checked_at text not null default current_timestamp
);
create index if not exists idx_boss_session_checks_time on boss_session_checks(checked_at);
```

### `selector_health`（选择器健康度）
```sql
create table if not exists selector_health (
  id text primary key,
  selector_key text not null,          -- recommend_candidate_card 等
  success integer not null default 0,
  fail integer not null default 0,
  last_error text,
  updated_at text not null default current_timestamp
);
create unique index if not exists idx_selector_health_key on selector_health(selector_key);
```

## 4. 接口清单（V2）

## 4.1 会话与登录
- `POST /api/boss/session/check`
  - 入参：无
  - 出参：`{ logged_in, reason, current_url, screenshot_path }`
- `POST /api/boss/session/save`
  - 入参：`{ wait_seconds? }`
  - 出参：`{ ok, login_detected, storage_state_path, screenshot_path }`

## 4.2 任务
- `POST /api/v2/recommend/tasks`
  - 入参：
    - `job_id`
    - `max_candidates`
    - `max_pages`
    - `sort_by`
    - `greet_threshold`
    - `greet_allow_non_recommend`
    - `search_config.keyword/city`
  - 出参：`{ task_id, status }`
- `POST /api/v2/tasks/{task_id}/start`
  - 功能：执行任务（先校验 session，再跑 recommend）
- `GET /api/v2/tasks/{task_id}`
  - 出参：任务详情 + token_usage + 统计字段
- `GET /api/v2/tasks/{task_id}/summary`
  - 出参：`processed/recommend/review/reject/greet_success/greet_failed`
- `GET /api/v2/tasks/{task_id}/logs`
  - 出参：结构化运行日志

## 4.3 候选人与复核
- `GET /api/v2/tasks/{task_id}/candidates`
  - 支持筛选：`decision`、`min_score`、`has_resume`、`has_greet_action`
- `GET /api/v2/candidates/{candidate_id}`
  - 出参：候选人结构化字段 + evidence + 打分细节
- `GET /api/v2/candidates/{candidate_id}/screenshot`
  - 出参：截图文件流
- `POST /api/v2/candidates/{candidate_id}/review`
  - 入参：`{ action: reviewed_completed, comment, final_decision }`
- `POST /api/v2/candidates/{candidate_id}/actions/greet/retry`
  - 入参：`{ reason }`
  - 出参：动作结果（success/failed + error）

## 4.4 运营指标
- `GET /api/v2/metrics/overview?from=...&to=...`
  - 出参：任务成功率、推荐率、打招呼成功率、Token、平均耗时
- `GET /api/v2/metrics/model-usage?from=...&to=...`
  - 出参：按模型统计 token、调用次数、失败率、P95 延迟

## 5. 任务流（标准执行链路）

## 5.1 主流程
1. HR 在任务页创建任务（岗位 + 人数 + 阈值 + 排序）。
2. 系统执行 `session/check`：
   - 若 `valid` -> 进入步骤 4
   - 若 `invalid` -> 步骤 3
3. 执行 `session/save`，人工登录后保存 `storage_state`。
4. 进入 `recommend` 页面，扫描候选人卡片并去重。
5. 逐个候选人执行：
   - 打开详情
   - 下载简历（失败则导出文本兜底）
   - 截图（建议全页）
   - LLM 抽取结构化字段
   - 评分卡判分
   - 满足阈值则打招呼
6. 写入候选人、快照、评分、动作、日志、token。
7. 汇总任务统计，输出到 HR checklist。

## 5.2 失败分支
- 会话失效：任务置为 `blocked_login_required`
- 页面结构变化：记录 `selector_health.fail + last_error`，任务置 `selector_error`
- 模型失败：保留 `fallback` 抽取并标记 `model_calls.status=fallback`
- 打招呼失败：记录 `candidate_actions.status=failed`，支持重试接口

## 6. 监控指标（可运营）

## 6.1 任务级指标
- `task_success_rate` = completed / started（目标 >= 95%）
- `task_p95_duration_minutes`（目标 <= 20 分钟，50 人）
- `session_valid_rate` = valid_checks / all_checks（目标 >= 90%）

## 6.2 数据质量指标
- `candidate_parse_success_rate` = 有效结构化字段 / 全量候选人（目标 >= 95%）
- `fallback_ratio` = fallback 抽取人数 / 全量候选人（目标 <= 20%）
- `selector_fail_rate` = selector_fail / selector_total（目标 <= 2%）

## 6.3 模型与成本指标
- `total_tokens`（按日/任务）
- `token_per_candidate_avg`（目标逐月下降）
- `model_call_error_rate`（目标 <= 3%）
- `model_call_p95_latency_ms`（目标 <= 8000ms）

## 6.4 业务效果指标
- `recommend_rate` = recommend / processed（用于调阈值，不设硬目标）
- `greet_success_rate` = greet_success / greet_attempted（目标 >= 90%）
- `hr_review_pass_rate` = HR 最终通过 / HR复核总数（用于校准评分卡）
