- 运维/实施人员：负责环境初始化、服务启动、插件安装、OCR 配置、数据备份与故障排查。

## 2. 第二阶段已交付能力

### 2.1 HR 侧能力

- `JD评分卡`：统一管理第一阶段内置评分卡与第二阶段自定义 JD 评分卡
- 根据 JD 文本自动生成评分卡
- 支持 HR 调整评分规则、阈值、硬性条件
- 支持批量导入 `pdf / doc / docx` 简历
- 批量导入后自动打分、筛查、输出推荐/复核/拒绝结果
- 扫描版 PDF 自动尝试 OCR
- `任务执行`、`推荐处理台`、`Checklist`、`高级搜索` 与 `JD评分卡` 已打通

### 2.2 插件与推荐流能力

- Chrome 插件自动识别 BOSS 候选人详情页
- 插件自动入库候选人并回填评分
- 推荐流任务与插件评分共用统一评分卡体系
- BOSS 登录改为：
  - HR 在已安装插件的 Chrome 中手动登录 BOSS
  - 插件自动同步当前浏览器会话
  - 平台只负责校验同步后的会话是否有效

### 2.3 运维侧能力

- 本地服务启动/停止/健康检查脚本
- SQLite 本地业务库
- 本地截图、简历、认证状态文件归档
- PaddleOCR 扫描件识别接入
- Chrome 插件支持本机与内网服务地址配置

## 4. HR 操作手册

## 4.1 首次使用前准备

### 步骤 1：登录后台

1. 打开 `http://127.0.0.1:8080/login`
2. 输入账号密码 `admin / admin`
3. 登录后默认进入 `任务执行`

### 步骤 2：安装或刷新 Chrome 插件

1. 打开 `chrome://extensions`
2. 打开“开发者模式”
3. 点击“加载已解压的扩展程序”
4. 选择目录 `chrome_extensions/boss_resume_score`
5. 如果插件已安装但代码有更新，点击一次“刷新”
6. 打开插件侧边栏，在“后端地址”填写：
   - 本机：`http://127.0.0.1:8080`
   - 内网：填写实际内网服务地址

### 步骤 3：手动登录 BOSS 并同步会话

当前推荐流程不再拉起单独的 BOSS 登录浏览器，正确方式如下：

1. 在已安装插件的 Chrome 中打开 BOSS 页面
2. HR 手动完成扫码登录
3. 登录完成后刷新一次 BOSS 页面
4. 回到 `任务执行` 页面
5. 点击“检查已同步的 BOSS 会话”
6. 页面提示通过后，再执行 Recommend 任务

如果会话异常，可先点击“清空已保存会话”，再重新登录并刷新 BOSS 页面。

## 4.2 `任务执行` 页面使用

页面位置：

`/hr/tasks`

适用场景：

- 校验当前 BOSS 会话是否可用
- 创建 Recommend 任务
- 配置岗位、最大人数、最大分页与排序方式

标准操作流程：

1. 先确认 BOSS 已在 Chrome 中手动登录
2. 点击“检查已同步的 BOSS 会话”
3. 选择岗位
4. 设定 `最大人数` 与 `最大分页`
5. 点击“创建并执行 Recommend 任务”
6. 任务完成后系统自动跳转到 `Checklist`

建议操作：

- 日常筛查先使用默认排序“活跃优先”
- 批量跑任务前先做一次会话检查，避免 BOSS 登录过期
- 查看右侧执行日志，确认是否出现“会话检测通过并已保存”

## 4.3 `JD评分卡` 页面使用

页面位置：

`/hr/phase2`

适用场景：

- 新建 JD 自定义评分卡
- 编辑第一阶段内置评分卡
- 批量导入简历并筛查打分

### A. 新建或编辑 JD 评分卡

1. 打开 `JD评分卡`
2. 选择现有评分卡，或新建自定义卡
3. 粘贴 JD 文本
4. 让系统自动生成评分卡
5. 根据岗位要求调整：
   - `hard_filters`
   - `thresholds`
   - `weights`
   - 关键字段命名与规则说明
6. 保存评分卡

说明：

- 第一阶段公式卡与第二阶段自定义卡现在已在统一列表管理
- 内置公式卡适合已有校准岗位
- 自定义卡适合新岗位快速试跑

### B. 批量导入简历

1. 在 `JD评分卡` 选择支持导入的 JD 卡
2. 上传 `pdf / doc / docx`
3. 等待系统解析与打分
4. 查看导入结果列表
5. 重点关注：
   - 总分
   - 推荐/复核/拒绝
   - 解析是否成功
   - OCR 是否触发

文件支持说明：

- 文本型 PDF：直接提取文本
- 扫描版 PDF：自动尝试 PaddleOCR
- Word：支持 `doc / docx`

导入失败时优先检查：

- 文件是否损坏
- PDF 是否为扫描件
- OCR 是否可用

## 4.4 `推荐处理台` 页面使用

页面位置：

`/hr/workbench`

适用场景：

- 查看插件入库与任务采集结果
- 保存阶段、原因码、标签和跟进动作
- 人工接管候选人状态

推荐使用方式：

1. 按来源筛选：
   - 插件入库
   - 任务采集
2. 按阶段筛选：
   - 已评分
   - 建议沟通
   - 待跟进
   - 已淘汰
3. 点击候选人查看右侧详情
4. 修改处理状态并保存

说明：

- 人工状态不会被自动评分覆盖
- 同名候选人会按身份键和最新记录做去重展示

## 4.5 `Checklist` 页面使用

页面位置：

`/hr/checklist`

适用场景：

- 查看某次任务的完整产出
- 复核每位候选人的评分结果与证据
- 回看截图、提取字段、模型解释和决策

建议重点查看：

- 候选人基础信息是否正确
- 结构化字段是否符合原简历
- 硬性过滤原因是否合理
- 最终分数与推荐结论是否符合岗位预期

## 4.6 `高级搜索` 页面使用

页面位置：

`/hr/search`

适用场景：

- 基于 JD 或自然语言需求检索本地简历库
- 快速查找已导入简历中的匹配候选人
- 配合 `JD评分卡` 形成“评分卡 + 搜索”闭环

常见检索方式：

- 关键词搜索：如 `Python Linux Redis 本科 3年`
- 自然语言搜索：如“找做过教育行业 Python 后端、有 Linux 和中间件经验的人”

## 4.7 Chrome 插件使用

标准流程：

1. 在 BOSS 打开候选人详情页
2. 打开插件侧边栏
3. 插件自动识别详情页并入库
4. 侧边栏显示 `JD 速览`
5. 后端返回完整评分
6. HR 在插件中保存跟踪状态

常见状态包括：

- 新入库
- 已评分
- 待复核
- 建议沟通
- 已沟通
- 待回复
- 待跟进
- 已邀约
- 面试已约
- 人才库
- 已淘汰
- 不再联系

## 5. 运维操作手册

```bash
bash scripts/bootstrap_phase1_env.sh
cp .env.local.example .env.local
```

建议检查：

- Python 版本 `>= 3.12`
- `.venv` 已创建
- Playwright Chromium 已安装
- 数据目录 `data/` 已生成

## 5.2 服务启动与停止

### 启动

```bash
bash scripts/start_phase1_server.sh
```

### 停止

```bash
bash scripts/stop_phase1_server.sh
```

### 健康检查

```bash
bash scripts/check_phase1_health.sh
```

健康检查至少要验证：

- `/health` 返回正常
- `/login` 可访问
- 插件目录存在
- 后端地址与端口配置正确

## 5.3 关键配置项

最低建议确认：

```bash
SCREENING_SERVER_HOST=127.0.0.1
SCREENING_SERVER_PORT=8080
SCREENING_PUBLIC_BASE_URL=http://127.0.0.1:8080
SCREENING_HEALTHCHECK_BASE_URL=http://127.0.0.1:8080
SCREENING_BROWSER_HEADLESS=false
SCREENING_BROWSER_STORAGE_STATE_PATH=/Users/jobs/Documents/CODEX/ZHAOPIN/data/auth/boss_storage_state.json
SCREENING_SEARCH_QDRANT_PATH=/Users/jobs/Documents/CODEX/ZHAOPIN/data/qdrant
```

第二阶段 OCR 相关配置见：

`.env.local.example`

重点前缀：

- `SCREENING_RESUME_OCR_PROVIDER`
- `SCREENING_RESUME_OCR_LANG`
- `SCREENING_RESUME_OCR_DET_MODEL_NAME`
- `SCREENING_RESUME_OCR_REC_MODEL_NAME`
- `SCREENING_RESUME_OCR_FORCE_ON_PDF`

## 5.4 OCR 安装与验证

扫描版 PDF 需要 PaddleOCR 支持。

建议安装顺序：

1. 安装 `paddlepaddle`
2. 安装 `paddleocr`
3. 重启后端服务
4. 用扫描版 PDF 做一次导入验证

建议验证点：

- 上传扫描版 PDF 后不再直接报“文本为空”
- 导入结果中能返回分数和决策
- 首次运行会自动下载 OCR 模型，后续会复用缓存

## 6. 常见问题处理

### 6.1 “检查已同步的 BOSS 会话”失败

排查顺序：

1. 确认 HR 已在 Chrome 中手动登录 BOSS
2. 刷新 BOSS 页面
3. 刷新插件
4. 检查 `data/auth/boss_session_meta.json` 是否更新
5. 如仍异常，点击“清空已保存会话”后重新登录

### 6.2 插件不显示结果

检查：

- 当前是否为候选人详情页
- 插件后端地址是否正确
- 后端服务是否正常
- 插件是否已刷新

### 6.3 批量导入扫描件 PDF 无结果

检查：

- PaddleOCR 是否已安装
- OCR 配置是否开启
- 首次模型下载是否成功
- PDF 是否损坏或清晰度过低

### 6.4 推荐任务无法执行

检查：

- BOSS 会话是否已通过检查
- Chrome 插件是否已同步最新会话
- 岗位配置是否有效
- BOSS 页面是否出现验证码、安全验证或风控提示
